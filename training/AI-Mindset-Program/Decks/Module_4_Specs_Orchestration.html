The role

The orchestrator

  • Decomposes work into steps an AI can execute.
  • Writes the specifications those agents consume.
  • Designs the verification gates between steps.
  • Owns the result.
From typist to conductor.

The new core skill

A prompt is a request.
A specification is a contract.

Goal

One sentence. Two goals = two specs.

Constraints

What must NOT change.

Acceptance criteria

Yes/no checks, not vibes.

Examples

One input→output pair.

Non-goals

The tempting adjacent thing — forbidden explicitly. Like briefing a contractor. Or a labrador.

Everything you’d tell a contractor. Written down. Once.

Worked example · documents

Spec for a tender-response section

# SPEC: Executive summary for tender [X] ## Goal One page the CEO can sign unedited. ## Constraints Max 400 words · only facts from our reference projects file · no promises we can’t evidence. ## Acceptance criteria [ ] Every requirement from the buyer’s checklist is addressed [ ] Every number appears verbatim in a source we hold [ ] Reads aloud in under 3 minutes ## Non-goals Do not invent partnership names. Do not restate the full company history — 2 lines max.
The buyer’s checklist is the acceptance test. Your Module 2 instinct (“extract requirements first”) — formalized.

Worked example · code

Spec for an agent-executed change

# SPEC: Add read-only status endpoint ## Context Legacy order service; consumers depend on current API behavior. ## Goal GET /orders/{id}/status returning cached state. ## Constraints Must not touch the public API surface · no new dependencies · follow existing controller conventions. ## Acceptance criteria [ ] All existing tests pass [ ] Returns 404 for unknown ids [ ] p95 < 50ms against the seeded fixture ## Non-goals No refactor of OrderRepository — tempting, forbidden, separate spec. ## Verification `npm test` + the 3 new cases above.

The integration problem

N tools × M systems = N×M custom integrations

Agents are only as useful as what they can reach. Without standards: the tangle that killed a decade of enterprise integrations. Two open protocols untangle it.

16 CONNECTORS. 16 THINGS TO BREAK.
ONE PROTOCOL. THE “USB-C MOMENT.”

Layer 1  FACT — public, checkable

MCP — Model Context Protocol

  • Open standard, introduced by Anthropic (Nov 2024); adopted across major vendors.
  • One protocol connecting AI applications to tools, data, systems — “USB-C for AI.”
  • Behaviorally: your agent reads your issue tracker, repo, database — through one standard socket, under least-privilege permissions you grant (Module 3).

Demo

Grounding, industrialized

  • An agent uses an MCP server to query a (fictional) issue tracker.
  • Output: a status report citing real ticket ids — every claim traceable.
  • Same rule as Module 3: no citation, no claim — now enforced by architecture.

Layer 2  FACT — public, checkable

A2A — Agent2Agent protocol

  • Open protocol, announced by Google (Apr 2025); now under Linux Foundation governance.
  • Agents discover and talk to other agents: publish capabilities (“agent card”), exchange tasks, cooperate across vendors.
  • MCP gives an agent hands. A2A gives agents colleagues.

Where orchestration fails

Error propagation

One bad extraction quietly poisons everything downstream — why gates sit between stages, not just at the end.

Context overflow

Very large systems force deliberate decomposition.

Black-box output

Stakeholders ignore what they can’t audit — the reason evidence tagging exists.

Over-engineering

A pipeline where one good prompt would do.

All four are mitigated by the same two disciplines: verification gates and evidence tagging.

Exercise · 60 minutes, pairs

Spec → agent → swap

  • Write a spec (template provided): code task on the sample repo — or a document task.
  • Run it: hand the spec verbatim to the agent. Don’t coach mid-run — let the spec do the work.
  • Review against your acceptance criteria — not against taste.
  • Swap specs with your pair and run theirs. Every “technically-compliant-but-wrong” result is an ambiguity — write it down. That list is the lesson.

Debrief

“Read us the spec line that allowed it”

  • Which ambiguities did the agent exploit?
  • Which acceptance criteria caught real problems?
  • What did the second iteration of the spec fix?
You just did the job. That was orchestration.
