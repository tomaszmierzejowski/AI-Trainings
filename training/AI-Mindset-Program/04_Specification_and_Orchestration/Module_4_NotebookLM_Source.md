# AI-Mindset Module 4 — From Writing Code to Writing Specifications: MCP, A2A, and Agent Orchestration (NotebookLM Source Document)

This document describes the professional shift from manual coding to specification design and agent orchestration. It defines the orchestrator role, teaches specification writing as the new core skill, explains the two interoperability standards that make multi-agent work practical — the Model Context Protocol (MCP) and the Agent2Agent protocol (A2A) — and grounds the whole argument in a verified production system: the author's multi-stage discovery pipeline for legacy software. Claims are marked VERIFIED (delivered work), FACT (public, checkable), or OPINION (the author's position, offered for argument).

## The provocation, honestly labeled

OPINION: code is becoming a commodity, and the specification is becoming the product. The module's framing analogy is Gutenberg: before print, the scribe's craft was penmanship; after print, writing mattered more than handwriting. Hand-crafted code stands to AI-generated code as illuminated manuscripts stand to printed books. The scribes did not disappear — the ones who became authors and publishers thrived. The program deliberately badges this as a position rather than a finding, because its own evidence standard requires marking where evidence ends and conviction begins.

What is not opinion is the inventory of skills that remain scarce when code generation becomes cheap: context (knowing what the business actually needs), judgment (knowing what good looks like), verification (proving a result correct — the subject of Module 3), and orchestration (designing which human, agent, or tool performs which step). None of these skills are typing.

## The orchestrator role and specification design

An orchestrator decomposes work into agent-executable steps, writes the specifications those agents consume, designs the verification gates between steps, and owns the final result — a shift the program summarizes as "from typist to conductor."

The core craft is specification design. A prompt is a request; a specification is a contract, with five parts: the goal, the constraints (what must not change), the acceptance criteria (how success is tested), worked examples, and explicit non-goals. The non-goals section is the part that most surprises practitioners, and it mirrors physical-world contracting exactly: a contractor is told what not to touch. The distinction matters because an agent given a one-line request will do something technically compliant with the words; an agent given a contract will do something compliant with the intent — and when it does not, the acceptance criteria catch it.

Specifications also exist in standing form: rules files such as CLAUDE.md, AGENTS.md, and .cursorrules, which every agent session in a repository inherits automatically. A rules file records project conventions, forbidden zones, and the commands used to verify changes — a project constitution readable by every agent the team will ever employ. Participants receive working templates of these files as takeaway artifacts, consistent with the program's fishing-rod philosophy: the deliverable is a reusable instrument, not a dependency on the trainer.

## MCP and A2A: the two interoperability layers

Agents are only as useful as what they can reach, and without standards, connecting N agent applications to M enterprise systems requires N×M custom integrations.

FACT: the Model Context Protocol (MCP) is an open standard introduced by Anthropic in November 2024 and subsequently adopted across major AI vendors. It defines one protocol through which an AI application connects to tools, data sources, and systems — the "USB-C for AI" analogy. Its behavioral meaning: an agent can read the team's issue tracker, repository, or database through a standard socket, under explicitly granted, least-privilege permissions — industrializing the grounding discipline taught in Module 3.

FACT: the Agent2Agent protocol (A2A) is an open protocol announced by Google in April 2025 and later placed under Linux Foundation governance. Where MCP connects an agent to tools, A2A connects agents to each other: an agent publishes its capabilities in an "agent card," and other agents can discover it, delegate tasks to it, and cooperate with it across vendor boundaries. The module's compression of the two layers: MCP gives an agent hands; A2A gives agents colleagues. In the resulting architecture, the human sits at the top as specifier and verifier, an orchestrating agent coordinates specialist agents over A2A, and each specialist reaches systems of record over MCP. The human does not manage the typing; the human manages the org chart.

## The production evidence: an orchestrated discovery pipeline

VERIFIED: the metrics quoted throughout this program were produced by an orchestrated agent pipeline in production use for legacy-system discovery. The pipeline stages specialized agents — ingestion, semantic analysis, security scanning, compliance mapping, synthesis — with human verification gates between stages. Its delivered results include: twenty legacy Java applications understood in three weeks, a 4–8× acceleration over conventional application comprehension with 187 endpoints mapped (Proof of Scale); over one hundred business rules extracted with source-traceable findings; and, on the reference engagement, approximately three weeks of orchestrated work replacing approximately fourteen weeks of traditional analysis at roughly 87 percent lower cost on the discovery and analysis phase only. The economics are the argument for the role: pipelines of this kind do not design, gate, or debug themselves.

The module is equally explicit about where orchestration fails, from field experience. Error propagation is the most dangerous mode: one faulty extraction early in a pipeline silently poisons every downstream agent, which is why verification gates sit between stages rather than only at the end. Context-window overflow constrains very large systems and forces deliberate decomposition. Black-box outputs are the leading adoption killer: enterprise stakeholders ignore technically correct results they cannot audit — the direct motivation for evidence tagging, in which every claim traces to a source line. And over-engineering is a constant temptation: a pipeline where a single well-written prompt would do. All four failure modes are mitigated by the same two disciplines: verification gates and evidence tagging.

## The career reframe and the 90-day path

The module's conclusion for practitioners: the professional role is not being replaced; it is being recompiled. Skills to stop over-investing in: syntax fluency, boilerplate speed, API memorization. Skills to double down on: domain context, specification writing, verification design, system decomposition, and agent-pipeline debugging. OPINION, labeled as such in delivery: the apprenticeship model for juniors moves up the stack — less syntax, more system thinking, earlier exposure to specification and review — rather than disappearing.

The module ends with a practiced exercise and a 90-day path. In the exercise, pairs write a specification for a task on a sample codebase, hand it to an agent, review the result strictly against the spec's acceptance criteria rather than taste, and then swap specifications and run each other's — a design that makes every ambiguity visible as concrete agent misbehavior. The 90-day path: in weeks one and two, adopt a rules file in the main repository; in weeks three through six, start every non-trivial task with a written specification; in weeks seven through twelve, build one two-stage pipeline with a verification gate around a real recurring task. The path is deliberately incremental: a pipeline built by someone who has not internalized specification writing is a very fast ambiguity amplifier.
