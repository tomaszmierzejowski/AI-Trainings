A public institute couldn't deploy software to its own hardware for 2 years. Nobody on the team could say why with certainty — the deployment pipeline had drifted out of anyone's full understanding.

We ran a full discovery pass first. It surfaced 200+ known CVEs the team hadn't been tracking. Then we went further: we read the git history the same way we read the code. Ten branches looked active. Only three were actually running in production.

We matched each of those three real branches to the specific hardware it was meant to deploy to, and automated the pipeline around that mapping.

The 2-year block ended. The institute could ship to its own hardware again — automatically, for the first time since the block began.

The blocker was never really in the code. It was in which version of the code anyone was actually looking at.

If your deployment has been stuck for longer than anyone wants to admit, start with one question: has anyone actually read the branch history, not just the current one?
